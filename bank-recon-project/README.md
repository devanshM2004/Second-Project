# Bank Reconciliation and Risk Detection Model

A Python portfolio project that simulates the daily work of a bank account analyst —
comparing bank records against internal system records, spotting discrepancies,
scoring risk, and generating reports.

I built this to practice data analysis with pandas and to demonstrate skills
relevant to entry-level roles in bank operations, account analysis, and financial
reconciliation.

---

## What problem does this solve?

Banks keep two separate records of every transaction:
1. **The bank's record** — the official file from the external bank or clearing house
2. **The internal system record** — what the bank's own software captured

These two should always match. When they don't, it could mean a processing error,
a missed posting, a duplicate charge, or something more serious. Reconciliation is
the process of finding and resolving those gaps before the books are closed.

This project automates that process end-to-end using Python and pandas.

## Pipeline overview

```
generate_data → reconcile → flag_risks → summary → report
```

### Step 1 — Generate data (`generate_data.py`)
Creates three CSV files with realistic-looking account and transaction data.
I also deliberately inject data quality problems so there's something
interesting to find:

| Problem | How many | What it tests |
|---|---|---|
| Missing transactions | 6 | Can we catch records the system never logged? |
| Amount mismatches | 8 | Do we spot when the amounts don't agree? |
| Date mismatches | 6 | Can we find records with the wrong posting date? |
| Duplicate transactions | 5 | Do we detect double-posted entries? |

### Step 2 — Reconcile (`reconcile.py`)
Merges the bank file and system file on `transaction_id` using a pandas
outer join, then classifies every row:

- `MATCHED` — both sources agree on amount and date
- `AMOUNT_MISMATCH` — same transaction ID, different dollar amount
- `DATE_MISMATCH` — same transaction ID, different posting date
- `BANK_ONLY` — bank has it, system doesn't
- `SYSTEM_ONLY` — system has it, bank doesn't

Also calculates a `variance` column (bank amount − system amount) so
the dollar exposure is visible on every row.

### Step 3 — Flag risks (`flag_risks.py`)
Scores each discrepancy using four simple rules and assigns a risk level:

| Rule | Logic | Points |
|---|---|---|
| Discrepancy type | BANK/SYSTEM_ONLY → +3, AMOUNT_MISMATCH → +2, DATE_MISMATCH → +1 | up to 3 |
| Variance size | ≥$10k → +3, ≥$1k → +2, ≥$100 → +1 | up to 3 |
| Duplicate flag | is_duplicate == True → +2 | 2 |
| Repeat account | 3+ issues this cycle → +2, 2 issues → +1 | up to 2 |

Score ≥ 6 = **HIGH** · Score 3–5 = **MEDIUM** · Score ≤ 2 = **LOW**

Each row also gets a plain-English `recommended_action` based on its
status and risk level.

### Step 4 — Summarise (`summary.py`)
Rolls everything up to one row per account showing total transactions,
discrepancy count and rate, total variance exposure, and the highest
risk level reached. Also generates a plain-text executive summary in
`summary_report.txt`.

### Step 5 — Report (`report.py`)
Filters to HIGH-risk rows only and saves them as `high_risk_transactions.csv`
— the shortlist an analyst would action first.
